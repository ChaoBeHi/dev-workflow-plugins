# 实现阶段插件

本插件覆盖 dev-workflow 的阶段 4-5：coding → scope-check。

## 能力需求

AI 应在运行时根据 system-reminder 中的可用技能列表，自动匹配以下能力对应的技能：

- **编码实现**：按方案编写代码，遵循项目规范
- **代码规范检查**：语言/框架专项规范、安全规范
- **变更范围对比**：计划 vs 实际 diff 分析

> 不硬编码任何具体 skill 名称。若本地无匹配技能，按本插件指令直接执行。

---

## 阶段 4: coding — 研发编码

**目标**：按确认的方案实现代码。

**AI MUST DO：**
- 严格按照 planing 确认的方案和用例编码。
- 不在此阶段"顺手"修改方案未涉及的文件或逻辑。

### 编码规范来源（优先级从高到低）

1. 项目级 `CLAUDE.md` / `AGENTS.md` 中的代码规范
2. 全局 `~/.claude/CLAUDE.md` / `~/.hermes/AGENTS.md` 中的代码规范
3. 语言/框架专项技能（若本地已安装且触发条件满足，优先调用）

### 行为约束

- 不引入方案未声明的依赖
- 不修改与需求无关的代码路径
- 不在此阶段做架构重构（应回退到 planing）
- 新增代码的日志、错误信息使用英文

---

## 阶段 5: scope-check — 范围校验

**目标**：确认实际产出与计划一致，无遗漏或越界。

**AI MUST DO：**

1. **范围检查**：对比 planing 方案，列出"计划写了但未实现"和"实现但计划未写"的项。
2. **差异说明**：解释每项差异的原因。
3. **刷新用例**：根据实际实现更新用例——代码行为可能与方案中的用例有偏差，修复前需和人类对齐。

**🚨Checkpoint**：
- 差异为 0 → 自动进入 testing。
- 有差异 → **停下来向人类报告差异清单**，等待人类决定：接受差异进入 testing / 打回 coding 修改。

### 差异报告格式

```
## Scope Check 报告

### 计划写了但未实现
- [ ] <项目> — 原因：<说明>

### 实现但计划未写
- [ ] <项目> — 原因：<说明>

### 结论
差异数: N | 建议: 接受进入 testing / 打回 coding
```
