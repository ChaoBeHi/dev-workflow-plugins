## PRD 范例

以下是一个完整 PRD 文档的参考结构：

```markdown
# PRD: 用户注销功能

- **状态**：待确认
- **来源**：AI 基于自然语言生成

## 功能概述
允许已登录用户主动注销当前会话，清除服务端 session 并作废 token，保护账户安全。

## 用户故事
- 作为已登录用户，我想要注销当前会话，以便防止他人在我离开后使用我的账户

## 验收条件
- Given 已登录用户 → When 点击注销 → Then session 被清除，客户端返回登录页
- Given 已注销用户 → When 在另一设备尝试访问 → Then 收到 401
- Given 未登录用户 → When 访问注销接口 → Then 返回 401（幂等）

## 业务规则与约束
- 注销仅影响当前设备 session，不影响其他设备的登录状态
- 注销后的 token 在剩余有效期内一律拒绝

## 非功能需求
- 性能：注销请求延迟 <200ms (P99)
- 安全：注销后 token 必须不可重用

## 范围边界
- 不做：全设备登出（仅单设备注销）
- 不做：注销审计日志（后续需求）

## 依赖与前置条件
- 已有登录认证模块（JWT + session 中间件）
- 无新增外部依赖
```

## 方案文档范例

以下是一个完整方案文档的参考结构：

```markdown
# 方案：用户注销功能

## 改动范围
- `src/auth/logout.ts` — 新增注销接口
- `src/middleware/session.ts` — 添加 session 清理逻辑
- `tests/auth/logout.test.ts` — 新增测试文件

## 核心变更

### 接口
- `POST /api/auth/logout` — 接收 refresh_token，清除服务端 session
- 返回值：`{ success: true }` 或 `{ error: "TOKEN_INVALID" }`

### 数据流
1. 前端调用 logout，传入 refresh_token
2. 服务端验证 token → 删除 session → 清除 cookie
3. 前端收到成功响应 → 清除本地 token → 跳转登录页

## 风险点与缓解

| 风险 | 缓解措施 |
|------|---------|
| token 已在别处被刷新导致当前 token 失效 | 忽略 token 验证失败，仍然清除客户端状态 |
| 并发请求在注销后仍携带旧 cookie | 中间件层添加 session 黑名单检查 |

## 依赖变更
无新增依赖。
```
