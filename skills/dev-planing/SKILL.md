# 规划阶段插件

本插件覆盖 dev-workflow 的阶段 1-3：input → context → planing。

## 能力需求

AI 应在运行时根据 system-reminder 中的可用技能列表，自动匹配以下能力对应的技能：

- **代码库检索与分析**：搜索关键符号、理解核心模块、定位接口和数据模型
- **方案/架构文档生成**：产出结构化方案文档
- **测试用例设计**：覆盖正常路径、边界条件、异常路径

> 不硬编码任何具体 skill 名称。若本地无匹配技能，按本插件指令直接执行。

---

## 阶段 1: input — 需求输入

| 角色 | 职责 |
|------|------|
| 人类 | 收集并输入需求 |
| AI | 用 1-2 句话**复述理解的需求**，确认与用户意图一致。若不明确，追问。 |

**🚨Checkpoint**：AI 复述需求后等待人类确认，确认无误进入 context。

---

## 阶段 2: context — 上下文装载

**目标**：在规划之前理解现有代码和约束。

**AI MUST DO：**

1. **代码库检索**：使用 `Grep` 搜索关键符号、`Read` 理解核心模块，定位与需求相关的现有接口、数据模型、配置。
2. **可行性校验**：判断需求是否与现有架构兼容，识别冲突点、缺失依赖、权限障碍。
3. **输出检索摘要**：列出涉及的文件/模块、潜在的冲突点。

**🚨Checkpoint**：
- 存在阻塞点 → **停下来向人类报告冲突原因**，等待决策（继续 / 调整需求 / 放弃）。
- 无阻塞点 → 自动进入 planing，不等待确认。

---

## 阶段 3: planing — 方案规划

**目标**：在写代码之前，对实现方案达成共识。

**AI MUST DO：**

1. **方案规划**：基于 context 的发现，提出实现方案。方案文档写入 `.workflow/plans/YYYY-MM-DD-<topic>-plan.md`。
2. **🚨方案评审**：以简洁格式呈现方案给人类，等待确认。人类可能要求修改——修改后重新提交。
3. **编写用例**：方案确认后编写测试用例，覆盖正常路径、边界条件、异常路径。写入 `.workflow/tests/cases/YYYY-MM-DD-<topic>-cases.md`。
4. **🚨用例评审**：提交用例给人类确认。

### 方案文档必须包含

- 改动范围（涉及文件/模块清单）
- 核心数据结构/接口变更
- 关键流程或算法说明
- 潜在风险点和缓解措施
- 依赖变更（如需要新引入库，必须声明并等待人类批准）

### 测试用例必须覆盖

- 正常路径：主流程 happy path
- 边界条件：空值、极限值、零值、重复输入
- 异常路径：网络失败、权限不足、数据损坏
- 并发/异步场景（如适用）
