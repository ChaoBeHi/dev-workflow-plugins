# 规划阶段插件

本插件覆盖 dev-workflow 的阶段 1-3：input → context → planing。

## 能力需求

根据 system-reminder 中的可用技能列表，自动匹配以下能力：

- **代码库检索与分析** — 搜索关键符号、理解核心模块、定位接口和数据模型
- **方案/架构文档生成** — 产出结构化方案文档
- **测试用例设计** — 覆盖正常路径、边界条件、异常路径

若本地无匹配技能，按本插件指令直接执行。不硬编码任何具体 skill 名称。

---

## 阶段 1: input — 需求输入与 PRD

**目标**：将人类需求转化为结构化的 PRD 文档，作为后续阶段的唯一需求锚点。

### 入口判断

人类可能提供两种形式的需求，AI 按路径分流：

```
人类输入
  ├── 指定了 PRD 文件/URL（如 "按 docs/prd/auth.md 开发"）
  │     → 路径 A：已有 PRD 文档，加载并提取
  │
  └── 自然语言描述（如 "帮我做一个用户注销"）
        → 路径 B：自然语言生成 PRD
```

两条路径汇聚到同一终点：`.workflow/prd/YYYY-MM-DD-<topic>-prd.md`。

### 路径 A：已有 PRD 文档

1. 使用 `Read` 加载 PRD 文件全文（或通过 MCP 工具读取外部协作工具的文档）
2. 提取关键验收条件，标注来源（原始 PRD 的章节/条目编号）
3. 按 PRD 模板格式写入 `.workflow/prd/`，在文档头部标注"来源：<原始 PRD 路径/URL>"
4. 若原始 PRD 中某些模板条目缺失，标注 `[待确认]` 并追问

### 路径 B：自然语言生成 PRD

1. **识别信息缺口**：对照 PRD 模板，判断人类描述中缺失的关键信息——功能模块边界、用户角色、核心流程、非功能约束。
2. **集中追问**：将关键问题一次性列出（不是逐条审问），让人类一次回答。追问聚焦在"不做就写不了 PRD"的核心缺口，不追问细节实现（那是 planing 的事）。
3. **按模板填充**：基于人类描述 + 追问回答，逐项填充 PRD 模板。能推断的内容直接填入，追问后仍不确定的**少量边缘项**标注 `[待确认]`。
4. 写入 `.workflow/prd/YYYY-MM-DD-<topic>-prd.md`

### PRD 模板

PRD 文档必须包含以下 8 项：功能概述、用户故事、验收条件（Given-When-Then）、业务规则与约束、非功能需求、范围边界、依赖与前置条件。完整范例见 `references/examples.md` 中的 PRD 范例。

### 🚨Checkpoint

AI 将 PRD 文档呈现给人类，人类确认（可能要求修改 → 改后重新确认）。确认通过后，PRD 锁定，进入 context。

## 阶段 2: context — 上下文装载

**目标**：在规划之前理解现有代码和约束。

**执行步骤：**

1. **代码库检索**：使用 `Grep` 搜索关键符号、`Read` 理解核心模块，定位与需求相关的现有接口、数据模型、配置。
2. **读取历史决策记录**：检查 `.workflow/prd/`、`.workflow/plans/`、`.workflow/knowledge/decisions/` 下是否有与当前需求相关的文档。目的是避免重复分析已被否决的方案，并理解现存设计的决策背景。
3. **可行性校验**：综合代码现状和历史决策，判断需求是否与现有架构兼容，识别冲突点、缺失依赖、权限障碍。
4. **输出检索摘要**：列出涉及的文件/模块、历史决策约束、潜在的冲突点。

**🚨Checkpoint**：
- 存在阻塞点 → **停下来向人类报告冲突原因**，等待决策（继续 / 调整需求 / 放弃）。
- 无阻塞点 → 自动进入 planing，不等待确认。

---

## 阶段 3: planing — 方案规划

**目标**：在写代码之前，对实现方案达成共识。

**执行步骤：**

1. **方案规划**：基于 context 的发现，提出实现方案。方案文档写入 `.workflow/plans/YYYY-MM-DD-<topic>-plan.md`。
2. **🚨方案评审**：以简洁格式呈现方案给人类，等待确认。人类可能要求修改——修改后重新提交。
3. **编写用例**：方案确认后编写测试用例，覆盖正常路径、边界条件、异常路径。写入 `.workflow/tests/cases/YYYY-MM-DD-<topic>-cases.md`。
4. **🚨用例评审**：提交用例给人类确认。

### 方案文档必须包含

- 需求溯源：列出方案中每个关键设计决策对应的 PRD 条目（如 "PRD 验收条件 3"），便于评审时逐条对账
- 改动范围（涉及文件/模块清单）
- 核心数据结构/接口变更
- 关键流程或算法说明
- 潜在风险点和缓解措施
- 依赖变更（如需要新引入库，必须声明并等待人类批准）

### 测试用例必须覆盖

- 正常路径：主流程 happy path
- 边界条件：空值、极限值、零值、重复输入
- 异常路径：网络失败、权限不足、数据损坏
- 并发/异步场景（如适用）
