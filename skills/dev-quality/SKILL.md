# 质量保障插件

本插件覆盖 dev-workflow 的阶段 6-8：testing → fixing → retest。

## 能力需求

AI 应在运行时根据 system-reminder 中的可用技能列表，自动匹配以下能力对应的技能：

- **测试执行与报告**：运行测试用例，产出结构化测试报告
- **缺陷根因分析**：区分实现缺陷与方案缺陷，定位根因
- **回归验证**：确认修复未引入新问题

> 不硬编码任何具体 skill 名称。若本地无匹配技能，按本插件指令直接执行。

---

## 阶段 6: testing — 测试验证

**目标**：验证实现是否符合用例预期。

**AI MUST DO：**
1. 运行 planing 阶段确认的测试用例。
2. 用例是否满足实际开发产物？不满足需回到 scope-check 刷新用例。
3. 产出测试报告，写入 `.workflow/tests/reports/YYYY-MM-DD-<topic>-test-report.md`。

### 测试报告必须包含

- 通过的用例列表
- 失败的用例列表（含失败原因）
- 测试覆盖率（如无工具支持，可联网检索相关产物并推荐给人类）
- 未覆盖的边界条件标注

---

## 阶段 7: fixing — 缺陷修复

**目标**：修复 testing 或 review 中发现的缺陷。

**AI MUST DO：**

### 缺陷分类（先分类，再修复）

| 类型 | 定义 | 处理方式 |
|------|------|----------|
| **实现缺陷** | 代码写错了，方案没问题 | 直接修复 |
| **方案缺陷** | 方案未覆盖该场景，或与 input 业务逻辑冲突 | 打回 planing，不在此阶段硬修 |

### 修复策略

| 修复规模 | 处理方式 |
|----------|----------|
| 轻量修复（≤1 文件且 ≤50 行，纯实现缺陷） | 跳过人类评审，直接修复 |
| 非轻量修复 | **🚨提交人类评审确认后再执行** |

### 缺陷报告必须标注

- 缺陷类型（实现 / 方案 / 与 input 业务逻辑冲突）
- 根因分析
- 修复方式
- 影响范围

---

## 阶段 8: retest — 回归测试

**目标**：验证修复没有引入新问题。

**AI MUST DO：**
1. 运行回归测试。
2. 若通过 → 产出测试报告（标注"第 N 轮修复"），进入 review。
3. 若不通过 → 分类处理：

| 失败原因 | 处理方式 |
|----------|----------|
| 实现缺陷 | 产出问题报告，打回 fixing |
| 方案缺陷 | 产出问题报告，打回 planing |

### 循环熔断

| 回退类型 | 上限 | 超限处理 |
|----------|------|----------|
| fixing 循环 | 3 轮 | 🚨暂停，向人类报告完整修复历史，等待决策 |
| planing 回退 | 1 次 | 🚨暂停，提示需求可能需回到 input 重新定义 |
