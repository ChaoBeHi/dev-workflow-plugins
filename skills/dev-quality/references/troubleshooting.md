# 质量保障故障恢复指南

## testing 阶段：测试环境不可用

**现象**：测试命令执行失败，报错与业务逻辑无关（如依赖缺失、端口占用、数据库连接超时）。

**处理**：
1. 区分环境故障 vs 代码故障——环境故障的特征：所有用例（包括无关用例）全部失败、报错信息指向基础设施而非断言
2. 环境故障 → 报告人类，提供修复建议（如 `npm install`、`docker-compose up`），等待环境就绪后重试
3. 代码故障 → 正常进入 fixing 流程

**示例报告**：

```
[环境故障] 所有 12 个测试用例均因 `ECONNREFUSED :5432` 失败。

PostgreSQL 未运行。请执行 `docker-compose up -d postgres` 后重新触发 testing。
```

## testing 阶段：用例过时

**现象**：用例期望值与当前方案/代码行为不一致，但代码行为是正确的（用例未随方案更新而更新）。

**处理**：
1. 确认过时原因——用例写于方案 v1，代码已按方案 v2 实现
2. 不修改代码去适配过时用例——修正用例以匹配当前方案
3. 标注用例变更，供人类确认

## fixing 阶段：根因定位困难

**现象**：缺陷可复现但无法快速定位根因。

**处理**：
1. 缩小范围：二分法注释/禁用代码块，确定最小复现单元
2. 添加诊断日志：在关键路径打点，输出中间状态
3. 若 3 次尝试仍未定位 → 🚨暂停，向人类报告已排查路径和剩余可疑点，请求补充上下文

**报告模板**：

```
[排查阻塞] 缺陷：并发注销后 session 未清除。

已排查：
- ❌ session 删除逻辑 — 代码正确，单测通过
- ❌ Redis 连接 — 正常读写
- ❓ 中间件执行顺序 — 疑似黑名单中间件在 session 中间件之后加载

请求确认：中间件加载顺序是否有显式配置？或是否能提供本地复现步骤？
```

## fixing 阶段：方案缺陷误判

**现象**：看起来是代码写错，实际上是方案未覆盖该场景。

**判定规则**：

| 判断条件 | 类型 | 处理 |
|----------|------|------|
| 方案中明确写了该场景的处理方式，代码未遵循 | 实现缺陷 | 直接修复 |
| 方案中未提及该场景，但现有架构可自然覆盖 | 实现缺陷 | 补充代码 |
| 方案中未提及该场景，且需要改变数据流/接口/架构才能覆盖 | **方案缺陷** | 打回 planing |
| 方案与 input 需求描述的业务规则冲突 | **方案缺陷** | 打回 planing |

**关键信号（方案缺陷）**：
- 修复需要修改方案文档中的接口签名或数据模型
- 修复需要引入方案未声明的新依赖
- 同一根因导致 ≥3 个用例失败（方案的某个假设不成立）

## retest 阶段：修复引入新缺陷

**现象**：修复后原失败用例通过，但新用例失败。

**处理**：
1. 确认新失败是否由本次修复引起——回退修复，重跑测试验证
2. 若确认是修复的副作用 → 分析修复是否可缩小影响范围（更精准的修改点）
3. 若缩小范围后仍引发连锁失败 → 标记为方案级问题，考虑打回 planing

## 循环熔断恢复

**现象**：fixing 循环达到 3 轮上限。

详细恢复流程参见全局 `references/troubleshooting.md` 的「fixing 阶段：循环超限」章节。

**本阶段特殊处理**：
- 熔断触发后，产出的修复历史报告中需额外标注每轮的「缺陷类型变化」——若三轮均为同一类型，强化对应建议（方案缺陷 → 强烈建议回退 planing，实现缺陷 → 强烈建议全面回归测试）
