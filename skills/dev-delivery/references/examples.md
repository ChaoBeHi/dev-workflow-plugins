## ADR 范例

以下是一个完整架构决策记录的参考结构：

```markdown
# ADR: 注销后使用 Redis 黑名单保证并发安全

- **日期**：2026-06-24
- **状态**：已采纳
- **背景**
  编码阶段发现：仅删除服务端 session 无法保证并发安全——若用户在注销的同时发起新请求，
  该请求可能在 session 删除前通过校验，导致已注销用户仍能访问受保护资源。

- **决策**
  注销后在 Redis 中写入 token 黑名单（key = `blacklist:<token_jti>`，TTL = token 剩余有效期）。
  中间件层在 session 校验前先检查黑名单，命中则直接返回 401。

- **备选方案**
  | 方案 | 优点 | 缺点 | 结论 |
  |------|------|------|------|
  | Redis 黑名单（本方案） | 即时生效，实现简单 | 需维护黑名单 TTL | ✅ 采纳 |
  | JWT 短有效期 + 无状态 | 无服务端状态 | 无法主动注销（需等 token 过期），用户体验差 | ❌ |
  | 内存 Set | 更快 | 多实例不同步，重启丢失 | ❌ |
  | 每次请求查 DB | 无需额外基础设施 | 性能开销大 | ❌ |

- **后果**
  - 正面：并发安全得到保证，注销即时生效
  - 正面：TTL 自动清理，无内存泄漏风险
  - 负面：Redis 多了一次查询（<1ms 影响可忽略）
  - 负面：若 Redis 不可用，需降级为放行（安全优先 → 改为拒绝？记录为后续议题）
```
