# 交付与沉淀插件

本插件覆盖 dev-workflow 的阶段 9-11：review → output → knowledge。

## 能力需求

AI 应在运行时根据 system-reminder 中的可用技能列表，自动匹配以下能力对应的技能：

- **代码评审**：设计/逻辑/安全三维审查
- **PR 创建与管理**：规范化提交与变更描述
- **知识归档**：设计决策记录与记忆更新

> 不硬编码任何具体 skill 名称。若本地无匹配技能，按本插件指令直接执行。

---

## 阶段 9: review — 代码评审

**目标**：在输出之前，对实现进行设计、逻辑、安全三维审查。

**AI MUST DO：**

| 维度 | 检查内容 |
|------|----------|
| **设计缺陷** | 方案是否正确落地？有无架构层面的妥协需要记录？ |
| **逻辑漏洞** | 边界条件是否完整覆盖？错误处理是否合理？并发/异步场景是否安全？ |
| **安全漏洞** | 是否存在 OWASP Top 10 风险（XSS、SQL 注入、敏感信息泄漏等）？输入校验是否完备？ |

产出评审报告，写入 `.workflow/review/YYYY-MM-DD-<topic>-review.md`。

**🚨Checkpoint**：
- 无严重问题 → 自动进入 output。
- 存在代码级缺陷 → 产出问题清单，打回 fixing。
- 存在设计级缺陷 / 安全漏洞 → **产出报告，停下来等待人类决策**（修复 / 接受风险 / 回退 planing）。

---

## 阶段 10: output — 落地产出

**目标**：交付最终产物。

**AI MUST DO：**
1. 完成最终代码落地。
2. 记录所有变更，提交 PR。
3. **CI 通过后，🚨提示人类确认合并**。AI 不自行执行 merge。
4. 产出最终报告，写入 `.workflow/reports/YYYY-MM-DD-<topic>-final.md`。

### 最终报告必须包含

- 需求简述
- 变更文件清单
- 方案链接（`.workflow/plans/`）
- 测试报告链接（`.workflow/tests/reports/`）
- 评审报告链接（`.workflow/review/`）
- 已知限制和后续建议

---

## 阶段 11: knowledge — 知识沉淀

**目标**：将本次经验固化为可复用知识。

**AI MUST DO：**
1. **决策记录**：归档关键设计决策、备选方案及选择理由。写入 `.workflow/knowledge/decisions/YYYY-MM-DD-<topic>-adr.md`。
2. **记忆更新**：将可复用的发现（新模块结构、配置约定、踩坑记录）写入项目持久记忆系统。

### ADR 模板

```markdown
# ADR: <决策标题>

- **日期**：YYYY-MM-DD
- **状态**：提议 / 已采纳 / 已废弃
- **背景**：<为什么需要做这个决策>
- **决策**：<选择了什么方案>
- **备选方案**：<还考虑了哪些方案>
- **后果**：<采纳后的正面/负面影响>
```
