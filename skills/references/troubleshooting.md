# 故障恢复指南

## context 阶段：检索无结果

**现象**：Grep/Read 未找到与需求相关的现有代码。

**处理**：
1. 扩大搜索范围——检查是否有类似功能用了不同命名（如 `auth` vs `session` vs `login`）
2. 确认是否属于全新模块——全新模块无需检索现有代码，直接标注"无现有实现，从零构建"
3. 若为第三方集成类需求，检索 `package.json` / `requirements.txt` 确认依赖是否存在

**确认后**：向人类报告"未找到相关现有实现"，询问是否需要扩大搜索范围或直接视为全新模块。

## context 阶段：架构冲突

**现象**：需求与现有架构设计矛盾（如要求 RESTful 但项目使用 GraphQL，或要求同步调用但项目采用事件驱动）。

**处理**：
1. 精确描述冲突点：需求声称 X，现有架构要求 Y
2. 给出两个选项：适配现有架构 vs 局部引入新模式（含工作量估算）
3. 不在此阶段自行裁决

**示例报告**：

```
[阻塞] 需求要求 RESTful API，但项目当前使用 GraphQL（参见 src/graphql/schema.ts）。

选项：
A) 用 GraphQL mutation 实现注销，保持一致 — 工作量 1h
B) 新增 REST 端点，混合模式 — 工作量 3h，增加维护复杂度

推荐 A。请确认。
```

## scope-check 阶段：差异过大

**现象**：实现与方案差异超过 5 项，无法快速判断是否合理。

**处理**：
1. 逐项标注差异类型：计划遗漏 / 实现越界 / 技术发现 / 需求变更
2. 对"技术发现"类差异——编码时发现方案未预料的技术约束——优先报告，这通常是方案缺陷的信号
3. 若差异多为"实现越界"，直接打回 coding 修剪

**决策树**：

```
差异类型？
├── 计划遗漏（方案明确写了，代码没做）
│   → 若 ≤2 项：补充实现，进入 testing
│   → 若 >2 项：打回 coding
├── 实现越界（方案没写，代码多做了）
│   → 若纯格式/辅助函数且无副作用：接受
│   → 若涉及业务逻辑：打回 coding，删除越界代码
├── 技术发现（编码时发现方案未考虑的技术约束）
│   → 🚨 暂停，报告人类。可能需回退 planing
└── 需求变更（人类在 coding 过程中口头追加）
    → 🚨 暂停，要求回退 input 重新定义需求
```

## fixing 阶段：循环超限

**现象**：fixing 循环达到 3 轮上限，熔断触发。

**处理**：
1. 列出 3 轮中每次修复的根因
2. 若同一 bug 重复出现 → 标记为方案缺陷，建议回退 planing
3. 若每次是不同的新 bug → 标记为实现质量问题，建议全面回归测试

**恢复步骤**：
1. 产出完整的修复历史报告（每轮：根因 + 修复 + 影响范围）
2. 🚨 提交人类决策：继续修复（重置计数）/ 回退 planing / 接受残余风险

## planing 回退：方案修正

**现象**：从 fixing/retest/review 阶段回退到 planing，方案需修正。

**处理**：
1. 不重写整个方案——做增量修正，保留已验证通过的部分
2. 修正后标记变更点（`[v2 新增]` / `[v2 修改]`），方便人类聚焦审阅
3. 已通过的测试用例如仍适用则保留，仅追加新用例

**增量修正范例**：

```markdown
## 方案：用户注销功能 (v2)

### [v2 新增] Session 黑名单机制
编码阶段发现：仅删除 session 不足以保证并发安全。
新增方案：注销后在 Redis 中写入 token 黑名单（TTL = token 剩余有效期）。

### [v2 修改] 接口返回值
原：`{ success: true }`
改：`{ success: true, blacklisted_until: "2026-06-24T12:00:00Z" }`
```
